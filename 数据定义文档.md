# 用户 / 角色 / 权限 数据定义文档

> 本文档依据前端 `cloudfrontproject` 的页面表单、列表列、Mock 数据（`src/types.ts`、`src/data.ts` 及 `src/components/*`）整理而成，供后端创建实体类与生成建表 SQL 使用。
>
> - **技术栈假设**：数据库以 **MySQL 8.0** 为例（可按需改为 PostgreSQL 等）；实体类以 **Java + Lombok** 为示例（可适配 Go / C# / Python / TS 等）。
> - **字段来源标注**：`【需求】` = 前端已有明确字段；`【建议】` = 前端未直接体现、但为后端常规做法的补充字段（审计、软删除等），可按需取舍。
> - 权限模型：**RBAC + 用户直接授权** 混合（角色↔权限 多对多；用户↔角色 多对多；用户↔权限 多对多，用于绕过角色直接授权）。

---

## 一、实体关系总览

```
        sys_user                    sys_role                 sys_permission
        (用户)                      (角色)                   (权限策略)
           |                           |                           |
           |  N..N                     |  N..N                     |
           +--- sys_user_role ----------+                           |
           |                                                        |
           +------------- sys_user_permission (用户直接授权) -------+  (N..N)
                                       |                           |
                                       +--- sys_role_permission ---+  (N..N)

        sys_operation_log (操作日志，弱关联 sys_user)
```

- `sys_user_role`：用户拥有的角色（用户加入角色后继承其全部权限）。
- `sys_role_permission`：角色绑定的权限。
- `sys_user_permission`：**直接授予用户**的权限（前端“权限设置”功能，独立于角色）。

## 二、实体定义

### 2.1 用户表 `sys_user`

| 字段 | SQL 类型 | 必填 | 默认 | 约束 | 说明 |
|---|---|---|---|---|---|
| id | BIGINT UNSIGNED | 是 | AUTO_INCREMENT | 主键 | `【建议】` 主键 |
| account | VARCHAR(64) | 是 |  | 唯一 | `【需求】` 用户账号，允许英文字母/数字/特殊符号（Mock 中为 11 位数字串） |
| name | VARCHAR(64) | 是 |  |  | `【需求】` 用户名称，允许中文/英文/数字/特殊符号 |
| phone | VARCHAR(20) | 否 |  |  | `【需求】` 手机号码，固定 11 位数字 |
| email | VARCHAR(128) | 否 |  |  | `【需求】` 电子邮箱 |
| avatar | VARCHAR(512) | 否 |  |  | `【需求】` 头像 URL（前端上传后存储） |
| password | VARCHAR(100) | 是 |  |  | `【建议】` 登录密码**哈希值**（如 BCrypt，禁止明文） |
| status | TINYINT | 是 | 1 |  | `【需求】` 状态：1=启用，0=禁用 |
| password_change_required | TINYINT | 否 | 0 |  | `【建议】` 重置/创建密码后是否要求下次登录修改：1=是，0=否 |
| created_at | DATETIME | 否 | CURRENT_TIMESTAMP |  | `【建议】` 创建时间 |
| updated_at | DATETIME | 否 | CURRENT_TIMESTAMP ON UPDATE |  | `【建议】` 更新时间 |
| deleted_at | DATETIME | 否 | NULL |  | `【建议】` 软删除时间（NULL=未删除） |

> 说明：创建用户时前端选择“默认密码/自定义密码”，重置密码时选择“默认/随机/自定义 + 生效时间(立即/次日) + 通知方式(短信/邮箱/其他) + 是否下次必须修改”——这些属于**接口请求参数**；其中“是否下次必须修改”建议落库为 `password_change_required`，“次日生效”可由定时任务或 `updated_at` 配合调度实现。

### 2.2 角色表 `sys_role`

| 字段 | SQL 类型 | 必填 | 默认 | 约束 | 说明 |
|---|---|---|---|---|---|
| id | BIGINT UNSIGNED | 是 | AUTO_INCREMENT | 主键 | `【建议】` 主键 |
| name | VARCHAR(64) | 是 |  | 唯一 | `【需求】` 角色名称（唯一） |
| description | VARCHAR(255) | 否 |  |  | `【需求】` 角色说明 |
| status | TINYINT | 是 | 1 |  | `【需求】` 角色状态：1=启用，0=禁用 |
| created_at | DATETIME | 否 | CURRENT_TIMESTAMP |  | `【建议】` |
| updated_at | DATETIME | 否 | CURRENT_TIMESTAMP ON UPDATE |  | `【建议】` |
| deleted_at | DATETIME | 否 | NULL |  | `【建议】` 软删除 |

### 2.3 权限（策略）表 `sys_permission`

| 字段 | SQL 类型 | 必填 | 默认 | 约束 | 说明 |
|---|---|---|---|---|---|
| id | BIGINT UNSIGNED | 是 | AUTO_INCREMENT | 主键 | `【建议】` 主键 |
| name | VARCHAR(64) | 是 |  | 唯一 | `【需求】` 权限名称，形如 `system1-email-readaccess` |
| description | VARCHAR(255) | 否 |  |  | `【需求】` 权限说明 |
| product_service | VARCHAR(64) | 否 |  |  | `【建议】` 所属产品服务（前端“新增权限”弹窗有该下拉项，Mock 未给出，建议补） |
| created_at | DATETIME | 否 | CURRENT_TIMESTAMP |  | `【建议】` |
| updated_at | DATETIME | 否 | CURRENT_TIMESTAMP ON UPDATE |  | `【建议】` |
| deleted_at | DATETIME | 否 | NULL |  | `【建议】` 软删除 |

> 权限命名规律（来自 Mock）：`{系统}-{能力}-{访问级别}`，例如 `system1 / email / readaccess`（只读）或 `fullaccess`（管理）。这是一种**策略式权限点**，而非传统的“资源:操作”细粒度，建议直接以 `name` 作为策略码下发与鉴权。

### 2.4 操作日志表 `sys_operation_log`

| 字段 | SQL 类型 | 必填 | 默认 | 约束 | 说明 |
|---|---|---|---|---|---|
| id | BIGINT UNSIGNED | 是 | AUTO_INCREMENT | 主键 | 主键 |
| operator_id | BIGINT UNSIGNED | 否 |  | 索引 | `【建议】` 操作人 ID（外键 sys_user.id，便于统计） |
| operator_name | VARCHAR(64) | 否 |  |  | `【需求】` 操作人名称（冗余，便于展示） |
| module | VARCHAR(64) | 否 |  |  | `【需求】` 操作模块：角色管理 / 用户管理 / 权限管理 / 日志管理 |
| action | VARCHAR(128) | 否 |  |  | `【需求】` 操作内容（如“创建角色”“批量授权”“重置用户密码”） |
| ip | VARCHAR(45) | 否 |  |  | `【需求】` IP 地址（支持 IPv6 用 45 位） |
| status | TINYINT | 否 | 1 |  | `【需求】` 操作状态：1=success，0=fail |
| operated_at | DATETIME | 否 | CURRENT_TIMESTAMP |  | `【需求】` 操作时间（对应前端 `time` 字段） |

## 三、建表 SQL（MySQL 8.0 / InnoDB / utf8mb4）

```sql
-- ========== 用户表 ==========
CREATE TABLE sys_user (
  id                     BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
  account                VARCHAR(64)     NOT NULL COMMENT '用户账号(唯一)',
  name                   VARCHAR(64)     NOT NULL COMMENT '用户名称',
  phone                  VARCHAR(20)     DEFAULT NULL COMMENT '手机号(11位)',
  email                  VARCHAR(128)    DEFAULT NULL COMMENT '电子邮箱',
  avatar                 VARCHAR(512)    DEFAULT NULL COMMENT '头像URL',
  password               VARCHAR(100)    NOT NULL COMMENT '密码哈希(BCrypt)',
  status                 TINYINT         NOT NULL DEFAULT 1 COMMENT '状态:1启用0禁用',
  password_change_required TINYINT       NOT NULL DEFAULT 0 COMMENT '是否要求下次登录改密:1是0否',
  created_at             DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_at             DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  deleted_at             DATETIME        DEFAULT NULL COMMENT '软删除时间',
  PRIMARY KEY (id),
  UNIQUE KEY uk_user_account (account)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';

-- ========== 角色表 ==========
CREATE TABLE sys_role (
  id          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
  name        VARCHAR(64)     NOT NULL COMMENT '角色名称(唯一)',
  description VARCHAR(255)    DEFAULT NULL COMMENT '角色说明',
  status      TINYINT         NOT NULL DEFAULT 1 COMMENT '状态:1启用0禁用',
  created_at  DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_at  DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  deleted_at  DATETIME        DEFAULT NULL COMMENT '软删除时间',
  PRIMARY KEY (id),
  UNIQUE KEY uk_role_name (name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='角色表';

-- ========== 权限(策略)表 ==========
CREATE TABLE sys_permission (
  id              BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
  name            VARCHAR(64)     NOT NULL COMMENT '权限名称/策略码(唯一)',
  description     VARCHAR(255)    DEFAULT NULL COMMENT '权限说明',
  product_service VARCHAR(64)     DEFAULT NULL COMMENT '所属产品服务',
  created_at      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_at      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  deleted_at      DATETIME        DEFAULT NULL COMMENT '软删除时间',
  PRIMARY KEY (id),
  UNIQUE KEY uk_permission_name (name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='权限策略表';

-- ========== 用户-角色 关联表 ==========
CREATE TABLE sys_user_role (
  user_id BIGINT UNSIGNED NOT NULL COMMENT '用户ID',
  role_id BIGINT UNSIGNED NOT NULL COMMENT '角色ID',
  PRIMARY KEY (user_id, role_id),
  KEY idx_role (role_id),
  CONSTRAINT fk_ur_user FOREIGN KEY (user_id) REFERENCES sys_user(id) ON DELETE CASCADE,
  CONSTRAINT fk_ur_role FOREIGN KEY (role_id) REFERENCES sys_role(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户-角色关联';

-- ========== 角色-权限 关联表 ==========
CREATE TABLE sys_role_permission (
  role_id       BIGINT UNSIGNED NOT NULL COMMENT '角色ID',
  permission_id BIGINT UNSIGNED NOT NULL COMMENT '权限ID',
  PRIMARY KEY (role_id, permission_id),
  KEY idx_permission (permission_id),
  CONSTRAINT fk_rp_role       FOREIGN KEY (role_id)       REFERENCES sys_role(id)       ON DELETE CASCADE,
  CONSTRAINT fk_rp_permission FOREIGN KEY (permission_id) REFERENCES sys_permission(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='角色-权限关联';

-- ========== 用户-权限 直接授权关联表 ==========
CREATE TABLE sys_user_permission (
  user_id       BIGINT UNSIGNED NOT NULL COMMENT '用户ID',
  permission_id BIGINT UNSIGNED NOT NULL COMMENT '权限ID',
  PRIMARY KEY (user_id, permission_id),
  KEY idx_permission (permission_id),
  CONSTRAINT fk_up_user       FOREIGN KEY (user_id)       REFERENCES sys_user(id)       ON DELETE CASCADE,
  CONSTRAINT fk_up_permission FOREIGN KEY (permission_id) REFERENCES sys_permission(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户-权限直接授权关联';

-- ========== 操作日志表 ==========
CREATE TABLE sys_operation_log (
  id            BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
  operator_id   BIGINT UNSIGNED DEFAULT NULL COMMENT '操作人ID',
  operator_name VARCHAR(64)     DEFAULT NULL COMMENT '操作人名称(冗余)',
  module        VARCHAR(64)     DEFAULT NULL COMMENT '操作模块',
  action        VARCHAR(128)    DEFAULT NULL COMMENT '操作内容',
  ip            VARCHAR(45)     DEFAULT NULL COMMENT 'IP地址',
  status        TINYINT         NOT NULL DEFAULT 1 COMMENT '状态:1成功0失败',
  operated_at   DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '操作时间',
  PRIMARY KEY (id),
  KEY idx_operator (operator_id),
  KEY idx_operated_at (operated_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='操作日志表';
```

> 说明：中间表的外键 `ON DELETE CASCADE` 可按团队规范改为应用层维护（删除业务对象时同步清理关联）。日志表不建议加外键约束（写入量大、避免级连锁）。

## 四、实体类示例（Java + Lombok，可适配其他语言）

```java
// ===== 用户 =====
@Data
public class User {
    private Long id;
    private String account;          // 唯一
    private String name;
    private String phone;            // 11位
    private String email;
    private String avatar;
    private String password;         // 哈希
    private Integer status;          // 1启用 0禁用
    private Integer passwordChangeRequired; // 1下次必须改密 0否

    // 关联（按需装载）
    private List<Role> roles;
    private List<Permission> permissions; // 用户直接授权

    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;
    private LocalDateTime deletedAt;
}

// ===== 角色 =====
@Data
public class Role {
    private Long id;
    private String name;             // 唯一
    private String description;
    private Integer status;          // 1启用 0禁用

    private List<User> users;            // 角色下成员
    private List<Permission> permissions; // 角色权限

    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;
    private LocalDateTime deletedAt;
}

// ===== 权限 =====
@Data
public class Permission {
    private Long id;
    private String name;             // 唯一，策略码，如 system1-email-readaccess
    private String description;
    private String productService;   // 所属产品服务

    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;
    private LocalDateTime deletedAt;
}

// ===== 操作日志 =====
@Data
public class OperationLog {
    private Long id;
    private Long operatorId;
    private String operatorName;
    private String module;           // 角色管理/用户管理/权限管理/日志管理
    private String action;
    private String ip;
    private Integer status;          // 1成功 0失败
    private LocalDateTime operatedAt;
}
```

> 关联表（`sys_user_role` / `sys_role_permission` / `sys_user_permission`）通常不单独建实体，用上面实体中的 `List` 关系 + DTO 即可；如需独立管理可再加 `UserRole` / `RolePermission` / `UserPermission` 三个仅含两个 id 字段的轻量实体。

## 五、业务操作清单（便于后端设计接口/服务方法）

| 模块 | 操作 | 涉及表 |
|---|---|---|
| 用户 | 新增 / 编辑 / 删除 / 启用 / 禁用 / 批量启用·禁用·删除 | sys_user |
| 用户 | 重置密码（默认/随机/自定义，生效时间，通知方式，是否下次改密） | sys_user |
| 用户-角色 | 给用户选择角色、移除角色 | sys_user_role |
| 用户-权限 | 给用户直接“添加权限/权限设置” | sys_user_permission |
| 角色 | 新增 / 编辑 / 删除 / 启用 / 禁用 / 批量操作 | sys_role |
| 角色-成员 | 角色成员管理（加/移用户） | sys_user_role |
| 角色-权限 | 角色权限管理（加/移权限） | sys_role_permission |
| 权限 | 新增 / 编辑 / 删除 / 批量删除 | sys_permission |
| 权限 | 批量授权（同时授权给多个角色和多个用户） | sys_role_permission, sys_user_permission |
| 日志 | 记录操作、按操作人/内容检索 | sys_operation_log |
| 登录 | 账号+密码登录 / 注册（账号、邮箱、密码） | sys_user |

## 六、设计备注

1. **权限计算**：用户最终权限 = 所有所属角色权限(`sys_user_role → sys_role_permission`) ∪ 用户直接权限(`sys_user_permission`)，取并集后去重。
2. **状态语义**：`status=0`(禁用) 的角色，其成员是否仍继承权限需产品确认（建议禁用角色的权限对成员立即失效）。
3. **密码安全**：仅存哈希；新建/重置密码的“默认密码/随机密码/自定义密码”为生成策略，结果统一哈希后入库；“次日生效”“短信/邮箱通知”为副作用，由服务层处理。
4. **产品服务**：前端“新增权限”有“产品服务”下拉，建议 `sys_permission.product_service` 单独成列，或拆出 `sys_product` 字典表关联（若产品服务需统一管理）。
5. **ID 策略**：此处用 BIGINT 自增；如团队偏好分布式可改雪花 ID（仍为 BIGINT）。
6. **软删除**：审计类（日志）表不做软删除；用户/角色/权限建议保留 `deleted_at` 支持回收站与唯一约束冲突处理。
